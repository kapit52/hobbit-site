# Ширский уголок — таверна для добрых путников

Полноценный сайт ресторана: PHP 8.2 + MySQL 8.0 + Docker.
Дизайн — Cinzel/Cormorant Garamond, пергаментная палитра, светлячки.

---

## Быстрый запуск через Docker

**Требования:** [Docker Desktop](https://www.docker.com/products/docker-desktop/) (Windows 10/11 с WSL2 или Mac).

```bash
docker compose up --build -d
```

| Адрес | Описание |
|-------|----------|
| http://localhost:8080 | Сайт таверны |
| http://localhost:8080/admin_login.php | Управление (для персонала) |

### Учётные данные

Вход для гостей — **по почте и паролю** (логин/никнейм убраны). Админ может войти и по почте, и по старому логину `admin`.

| Роль | Почта (или логин) | Пароль |
|------|-------------------|--------|
| Администратор | `admin@shire-corner.local` (или `admin`) | `shire123` |
| Демо-гость | `guest@example.com` | `shire123` |

Страница входа админа: `admin_login.php`. Гости входят на `login.php`.

### Остановка

```bash
docker compose down          # остановить, данные сохраняются
docker compose down -v       # полная очистка БД (начать с нуля)
```

## База данных и её хранение

**Данные БД хранятся в именованном Docker-томе `shire_db_data`** (см.
`docker-compose.yml`), а не в папке проекта. Так надёжнее на Windows: файлы БД
лежат внутри Docker и их нельзя случайно удалить из проводника «из-под»
работающего MySQL (именно это ломает базу). Что важно знать:

- ✅ `docker compose down` **сохраняет** базу — том остаётся, при следующем `up`
  все данные на месте.
- ⚠️ `docker compose down -v` **стирает** том — база при следующем старте
  пересоздаётся «с нуля» из снимка `docker/init/01_database.sql`.

Первый запуск (и любой запуск после `down -v`) наполняет БД из снимка
`docker/init/01_database.sql` (полная схема + данные: меню, фото-привязки слотов,
отзывы, аккаунты). Никаких ручных шагов не нужно.

### Перенос проекта на другой ПК

Источник правды о данных — снимок `docker/init/01_database.sql` (он в репозитории).
Том `shire_db_data` между машинами НЕ переезжает. Склонируй проект и запусти:
```
docker compose up --build -d
```
БД соберётся из снимка автоматически. Вместе с проектом едут:
- `public/uploads/` — сами файлы фотографий;
- `docker/init/01_database.sql` — снимок БД со всеми привязками фото.

### Полный сброс базы «с нуля»

```
docker compose down -v       # остановить и удалить том с данными
docker compose up -d         # БД пересоберётся из docker/init/01_database.sql
```

### Снимок-сид: сохранить правки из админки в репозиторий

`docker/init/01_database.sql` — сид для чистого старта. После изменений в админке
(загрузил/назначил фото, добавил блюда) **обнови снимок при поднятом стеке** — иначе
на свежем клоне этих правок не будет:
```
docker/snapshot-db.ps1      # Windows
sh docker/snapshot-db.sh    # Linux/Mac
```
Затем закоммить `docker/init/01_database.sql` и запушь.

## Тесты

Юнит-тесты на **PHPUnit 10** (без Composer, через `tests/tools/phpunit.phar`).
Покрывают «чистые» функции-хелперы — телефоны, цены, скидки, статусы, слоты брони.

```bash
# один раз скачать PHPUnit:
#   PowerShell: Invoke-WebRequest https://phar.phpunit.de/phpunit-10.5.phar -OutFile tests/tools/phpunit.phar
# прогон (стек поднят):
docker compose exec web php tests/tools/phpunit.phar --testdox
```

Текущий результат: `OK (47 tests, 49 assertions)`. Подробнее — в
**[tests/README.md](tests/README.md)** и в разделе 17 документации.
